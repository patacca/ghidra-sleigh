# ghidra-sleigh Design

## Problem

The Ghidra C++ decompiler requires compiled Sleigh specification files (`.sla`)
at runtime to decode instructions for each target ISA. These files are compiled
from `.slaspec` source files using the `sleighc` compiler, which is itself built
from Ghidra's C++ sources.

The compiled `.sla` files for all Ghidra-supported processors total 100-200 MB.
Bundling them inside the main `flatline` wheel would make it unacceptably large.

## Solution: separate runtime-data package

`ghidra-sleigh` is a standalone pip-installable package that:

1. Builds the `sleighc` compiler from Ghidra C++ sources at build time.
2. Compiles `.slaspec` files for the most important ISAs (x86, AArch64, RISC-V, MIPS — each in both 32- and 64-bit variants).
3. Ships the compiled `.sla` plus supporting files (`.ldefs`, `.pspec`, `.cspec`)
   as Python package data.
4. Exposes a single Python function `get_runtime_data_dir()` that returns the
   absolute filesystem path to the installed data root.

## Interface contract

### Python API

```python
ghidra_sleigh.get_runtime_data_dir() -> str
```

Returns the absolute path to a directory with this layout:

```
data/
  Ghidra/
    Processors/
      DATA/data/languages/
        data.ldefs, data.pspec, *.cspec, *.sla
      x86/data/languages/
        x86.ldefs, x86.pspec, x86-64.pspec, *.cspec, x86.sla, x86-64.sla
      AARCH64/data/languages/
        AARCH64.ldefs, AARCH64.pspec, *.cspec, AARCH64.sla
      RISCV/data/languages/
        riscv.ldefs, *.pspec, *.cspec, riscv.ilp32d.sla, riscv.lp64d.sla
      MIPS/data/languages/
        mips.ldefs, *.pspec, *.cspec, mips32le.sla, mips32be.sla, ...
```

This layout is consumed by `SleighArchitecture::scanForSleighDirectories()`
in the Ghidra decompiler C++ code. That function:
1. Looks for a `Ghidra/` directory within the root path (up to depth 2).
2. Under that, looks for `Processors/*/data/languages/` directories.
3. Scans those directories for `.ldefs` files to discover available languages.
4. Uses `.sla`, `.pspec`, `.cspec` files referenced by `.ldefs` entries.

### Version metadata

```python
ghidra_sleigh.GHIDRA_TAG       # str: Ghidra release tag
ghidra_sleigh.GHIDRA_COMMIT    # str: upstream commit hash
ghidra_sleigh.PROCESSORS       # tuple[str, ...]: included processor families
ghidra_sleigh.__version__      # str: package version
```

### File types shipped

| Extension | Purpose | Source |
|-----------|---------|--------|
| `.sla` | Compiled Sleigh specs (instruction decoding) | Generated by `sleighc` at build time |
| `.ldefs` | Language definitions (XML, maps language IDs to .sla/.pspec/.cspec) | Copied from upstream |
| `.pspec` | Processor specs (register definitions, address spaces) | Copied from upstream |
| `.cspec` | Compiler specs (calling conventions, data type mappings) | Copied from upstream |

### File types NOT shipped

| Extension | Reason excluded |
|-----------|----------------|
| `.slaspec` | Build-time source only; not needed at runtime |
| `.sinc` | Sleigh include files; build-time only |
| `.opinion` | Ghidra Java framework only |
| `.dwarf` | Ghidra Java debug info only |
| `.gdis` | Ghidra Java disassembler config only |
| `.register.info` | Ghidra Java register info only |

## Integration with flatline

### Current (MVP)

`flatline` declares `ghidra-sleigh` as a required dependency in
`pyproject.toml`. At runtime, when `runtime_data_dir` is not explicitly provided
by the user, `flatline` calls `ghidra_sleigh.get_runtime_data_dir()` to
discover the default data path.

```python
# In flatline._bridge or flatline._session (internal)
def _default_runtime_data_dir() -> str | None:
    try:
        import ghidra_sleigh
        return ghidra_sleigh.get_runtime_data_dir()
    except ImportError:
        return None
```

### Post-MVP (planned)

Make `ghidra-sleigh` an optional dependency. Users who provide their own
`.sla` files via `runtime_data_dir=` parameter do not need this package:

```toml
# pyproject.toml (post-MVP)
[project.optional-dependencies]
sleigh = ["ghidra-sleigh>=0.1.0"]
```

```bash
pip install flatline              # No bundled .sla, user provides own
pip install flatline[sleigh]      # Includes ghidra-sleigh
pip install ghidra-sleigh         # Standalone data package
```

This decision is tracked for post-MVP evaluation.

## Build process

```
[Ghidra C++ sources] --meson--> sleighc (executable)
                                    |
[.slaspec files] ----sleighc -a---> [.sla files]
                                    |
[.ldefs + .pspec + .cspec] --------+---> installed as package data
```

### Meson targets

1. **`sleighc`** (`executable`): Sleigh compiler built from CORE + SLEIGH +
   SLACOMP source sets (matching upstream `Makefile` partitioning).

2. **`sleigh-<PROC>`** (`custom_target` per processor): Runs
   `sleighc -a <processor_dir>` to compile all `.slaspec` files under that
   processor. Produces a `.stamp` file for dependency tracking.

3. **`install_subdir`** (per processor): Installs the
   `<PROC>/data/languages/` directory (including the just-compiled `.sla`
   files) into the package data tree.

### Build options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `ghidra_root` | string | `../third_party/ghidra` | Path to Ghidra source tree |
| `all_processors` | boolean | `false` | Compile all processors (not just priority) |

## ISA packaging strategy (planned)

The current default build includes only the most important ISAs (x86, AArch64, RISC-V, MIPS).
A decision is pending on how to expose different coverage levels to consumers:

- **pip extras** — `pip install ghidra-sleigh[light]` (major ISAs only) vs
  `pip install ghidra-sleigh[full]` (all 60+ Ghidra processors).
- **Separate packages** — `ghidra-sleigh-light` and `ghidra-sleigh-full` as independent
  distributions.

The choice depends on how well pip extras integrate with binary wheel selection and how large
the full-processor data set grows per platform.

## Versioning strategy

- Package version is independent of `flatline` version.
- Each release is pinned to exactly one Ghidra upstream commit.
- Ghidra upstream bump requires a new `ghidra-sleigh` release.
- `flatline` pins a minimum `ghidra-sleigh` version that matches its
  upstream pin (e.g., `ghidra-sleigh>=0.1.0` when both track
  `Ghidra_12.0.3_build`).

## Risks and mitigations

| Risk | Mitigation |
|------|-----------|
| Package size for all processors exceeds limits | Default build includes only priority ISAs (~30-80 MB); full build is opt-in |
| `.sla` binary format changes between Ghidra versions | Package is pinned to one Ghidra version; format changes require new release |
| Cross-processor `.cspec` references break isolated builds | DATA processor is always compiled first and included in every build |
| Build requires C++ compiler and zlib | Same requirement as flatline native bridge; documented in README |
